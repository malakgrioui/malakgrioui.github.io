GMT Data Backbone

Designed the core data infrastructure for Global Mangrove Trust — a scalable backbone enabling consistent, reliable data flow across field operations, satellite monitoring, and conservation reporting.

Organisation

Global Mangrove Trust is a mission-driven non-profit using technology and data science to drive large-scale environmental restoration. As the organisation scaled its operations across multiple geographies, it needed a robust, unified data foundation to connect field teams, satellite data, and decision-makers.

Context & Challenge

GMT's data was fragmented across tools and teams: field data collected via Kobotoolbox, satellite analysis outputs from Forest Scan, manually maintained reports, and stakeholder dashboards that were disconnected from live data. The challenge was to design a coherent backbone that unified these sources, ensured data integrity, and made information accessible to both technical teams and non-technical stakeholders in real time.

Key Contributions

  • Architecture Design Mapped existing data flows and designed a unified backbone connecting field collection, remote sensing, and reporting layers.
  • Data Model Definition Defined standardised schemas and data models to ensure consistency across sources — field surveys, satellite outputs, and carbon credit data.
  • Integration Planning Specified integration points between Kobotoolbox, Google Cloud Platform, Forest Scan outputs, and reporting interfaces.
  • Scalability Design Designed the infrastructure to scale across new geographies and project types without requiring architectural rework.
  • Stakeholder Alignment Facilitated workshops with technical leads, field teams, and management to validate the design against operational realities.
  • Documentation Produced full technical documentation and data flow diagrams to support implementation and onboarding.

Methodology & Tools

  • Approach Design thinking combined with iterative stakeholder validation cycles.
  • Diagramming Miro, Figma
  • Cloud Infrastructure Google Cloud Platform
  • Data Collection Layer Kobotoolbox
  • Collaboration Google Workspace

Lessons Learned

  • Data infrastructure design is fundamentally a communication problem — the hardest part is aligning diverse stakeholders on a single model of reality before writing a single line of code.
  • Standardising schemas early is the highest-leverage investment in a data system — it reduces rework downstream more than any other single decision.
  • Field realities always complicate clean architectures — designing in flexibility for partial data, connectivity gaps, and varied device types is not optional in conservation contexts.
